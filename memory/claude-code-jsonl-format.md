# Claude Code JSONL session format

Reference for `~/.claude/projects/<encoded-cwd>/<sessionId>.jsonl`. Each line is one JSON event in a Claude Code session. Used by [vault-mem-namespace](vault-mem-namespace.md) to convert sessions into `~/vault/chats/code/` markdown.

## Path encoding

`/Users/axinvd/projects/myProject` â†’ `~/.claude/projects/-Users-axinvd-projects-myProject/`. Forward slashes become hyphens; the leading `/` becomes a leading `-`. The last segment of the encoded directory name is what we use as the `project:` tag in chat archives.

Subagent sessions can nest deeper, e.g. `~/.claude/projects/-Users-axinvd-Documents-projects-someAgent/<parent-uuid>/subagents/agent-<id>.jsonl`. Our converter takes the immediate parent-dir basename as the project, which means subagents from project X are tagged `project: subagents` rather than `project: X`. Acceptable trade â€” tagging by spawning project would require following `parentUuid` chains.

## Event types (`type` field)

| Type | Meaning | Kept in markdown? |
|---|---|---|
| `user` | User prompt (when content is string or has no `tool_result` blocks) OR tool result (when `toolUseResult` key is present, or content list has `tool_result` blocks) | Real prompts only |
| `assistant` | Model response with `message.content` as list of blocks | Only `text` blocks |
| `system` | System message / hook injection | No |
| `attachment` | File attachments | No |
| `file-history-snapshot` | Code state snapshot | No |
| `permission-mode` | Bookkeeping | No |
| `last-prompt` | Bookkeeping | No |
| `progress` | Agent progress events | No |

## Content blocks (inside `message.content` arrays)

| Block type | Meaning | Kept? |
|---|---|---|
| `text` | Visible model text | Yes (assistant only) |
| `thinking` | Internal reasoning | No |
| `tool_use` | Tool invocation with `input` args | No |
| `tool_result` | Output from a tool (returned as `user` event) | No |

Including `tool_use.input` would balloon archive ~10Ă— with mostly noise (bash invocations, file paths). We chose text-only â€” same as `claude-conversation-extractor` does.

## Common event fields

- `uuid` â€” event ID
- `parentUuid` â€” predecessor event in the session graph
- `isSidechain` â€” true for events generated by subagents/sidechains
- `timestamp` â€” ISO format with milliseconds and Z, e.g. `2026-04-20T15:08:41.398Z`
- `userType` â€” distinguishes external/internal user events
- `sessionId` â€” present on `permission-mode` and `last-prompt` events; equals the JSONL filename stem
- `requestId` â€” on `assistant` events, ties to a model request

## Distinguishing real user prompts from tool results

Both arrive as `type: user`. Heuristic that works:

```
def is_real_user_prompt(event):
    if event["type"] != "user": return False
    if event.get("toolUseResult") is not None: return False
    content = (event.get("message") or {}).get("content")
    if isinstance(content, list):
        for b in content:
            if isinstance(b, dict) and b.get("type") == "tool_result":
                return False
    return bool(extract_text(content).strip())
```

`toolUseResult` is the most reliable marker. The list-of-blocks check catches edge cases where the field is absent.

## Identifying the current live session

Two complementary mechanisms:

**`~/.claude/sessions/<pid>.json`** â€” written by the running `claude` process. Schema:
```json
{
  "pid": 34042,
  "sessionId": "178f0d09-7646-4c41-bf4c-05d29b2f14fe",
  "cwd": "/Users/axinvd/projects/myProject",
  "startedAt": 1777125072662,
  "version": "2.1.119",
  "kind": "interactive",
  "status": "busy",
  "updatedAt": 1777129735339,
  "bridgeSessionId": "session_01Q3..."
}
```

**Process-tree walk.** From a script invoked inside a slash command, walk `getppid()` upward until a PID matches `~/.claude/sessions/<pid>.json`. That PID is `claude` itself; the file gives the active `sessionId`. Then `find ~/.claude/projects -name "<sessionId>.jsonl"` locates the JSONL.

This is the deterministic detection used in `~/scripts/claude_to_obsidian.py --current`. mtime-based "newest JSONL" is the fallback â€” mtime fails when:

- two `claude` instances run in parallel terminals (newest mtime is the other one's last turn)
- subagents write concurrently to their own JSONLs (could grab a subagent's file)

## Resume semantics

`claude --resume <session-id>` continues writing to the **same** JSONL â€” the sessionId stays stable. Implication: re-running `/mem:save` in a resumed session updates the same chat archive file (idempotent by UUID).
